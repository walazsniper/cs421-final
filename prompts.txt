  Question 1: VBool vs VPack for booleans                                                                                                                                                                                                                                     
                                                                                                                                                                                                                                                                              
  VPack (booleans share representation with constructors)                                                                                                                                                                                                                     
                                                                                                                                                                                                                                                                              
  Pros                                                                                                                                                                                                                                                                        
  - One uniform code path for case — comparison results can be cased on with the same machinery as user sum types.
  - Matches the IFLT tutorial convention; if you ever want to extend the project (e.g., booleans defined in user code as bool ::= False | True), there's no conversion layer.                                                                                                 
  
  Cons (these are the bug magnets)                                                                                                                                                                                                                                            
  - Magic tag numbers. You hardcode "True is tag 2, False is tag 1" (or the reverse) in the comparison primitives, in &/|, and in any printer. Off-by-one bugs here are silent — they don't crash, they just give the wrong answer.
  - Collisions with user type decls. If nextTag starts at 1 and your comparisons hardcode Pack{1,0} / Pack{2,0}, then the first user-declared 0-arity constructors collide with True/False. You'd have to either pre-seed ConCtx with reserved tags 1 and 2 (and bump nextTag 
  to 3), or emit a sentinel high tag for booleans. Either workaround is a place to forget something.                                                                                                                                                                         
  - Not actually exercised by any test. Nothing in parser-tests/ cases on a comparison result, so the "uniformity" win is theoretical.                                                                                                                                        
                                                                                                                                      
  VBool Bool (dedicated)                                                                                                                                                                                                                                                      
                                                                                                                                                                                                                                                                              
  Pros                                                                                                                                                                                                                                                                        
  - Haskell's type checker catches bugs. (&&) :: Bool -> Bool -> Bool — if you accidentally pass a VInt to &, GHC won't even compile it.                                                                                                                                      
  - No magic tag numbers anywhere.                                                                                                                                                                                                                                            
  - No collision with user constructor numbering — nextTag can start at 1 cleanly.
  - Printing is trivial: show True / show False.                                                                                                                                                                                                                              
                                                                                                                                                                                                                                                                              
  Cons
  - Two value worlds (booleans and packs) that don't compose. If a user ever writes case (x < y) of <1> -> ... ; <2> -> ..., it won't work — but no spec example or test does this.                                                                                           
  - One small mismatch with the IFLT convention.                                                                                                                                                                                                                              
                                                                                                                                                                                                                                                                              
  My recommendation: VBool. Lower bug surface, Haskell catches mistakes for you, and the only real downside is an edge case the tests don't cover.                                                                                                                            
                                                                                                                                                                                                                                                                              
  ---                                                                                                                                                                                                                                                                         
  Question 2: EBinop constructor vs primitive closures in initial env                                                                                                                                                                                                         
                                                                     
  EBinop Name (added AST constructor)
                                                                                                                                                                                                                                                                              
  Pros
  - Operators are syntactically distinct from variables in the AST. When you read a Show'd expression while debugging, EBinop "+" is unmistakable; EVar "+" looks like a typo.                                                                                                
  - The eval rule is one line: eval (EBinop n) _ = VPrim n [].                                                                                                                                                                                                                
  - No initial-env construction step — run doesn't have to seed anything.
                                                                                                                                                                                                                                                                              
  Cons                                                                                                                                                                                                                                                                        
  - One extra Expr constructor (allowed by spec, but more surface area).                                                                                                                                                                                                      
  - One extra eval case (trivial).                                                                                                                                                                                                                                            
                                  
  Primitive closures in env (parser emits EVar "+")                                                                                                                                                                                                                           
                                                                                                                                                                                                                                                                              
  Pros                                                                                                                                                                                                                                                                        
  - Zero new AST constructors — Expr stays exactly as the spec gave you.                                                                                                                                                                                                      
  - Uniform eval: + flows through the same EVar lookup as everything else.                                                                                                                                                                                                    
  
  Cons                                                                                                                                                                                                                                                                        
  - You have to build and thread an initial env containing "+", "-", ... everywhere — run seeds it, letrec has to union with it, supercombinator setup has to start from it. Forgetting to thread it in one place means + mysteriously becomes "undefined variable."
  - Operator names live in the same namespace as user variables. The parser prevents collisions today (lowercase-led identifiers can't contain +), but it's an invariant you have to keep mental track of.                                                                    
  - When debugging, EVar "+" blends in with user variables — easy to misread.                                                                                                                             
                                                                                                                                                                                                                                                                              
  My recommendation: EBinop. The "extra constructor" cost is tiny, and the "must remember to thread initial env" failure mode of the alternative is exactly the kind of bug that'd take an hour to track down later. EBinop keeps the operator world isolated and obvious.

      51 +## How it works                                                                                                                                                                                                      
      52 +                                                                                                                                                                                                                     
      53 +The pipeline is read → parse → evaluate → print:                                                                                                                                                                     
      54 +                                                                                                                                                                                                                     
      55 +1. **Parse** (`src/Parse.hs`) — Parsec turns source into a `HashMap Name Decl`.                                                                                                                                      
      56 +   Algebraic type declarations are stripped at parse time and their                                                                                                                                                  
      57 +   constructors registered in user state with fresh tags, so `Just`, `None`,                                                                                                                                         
      58 +   etc. become `EPack tag arity` nodes in the AST.                                                                                                                                                                   
      59 +2. **Evaluate** (`src/Interp.hs`) — a value-based interpreter. Closures                                                                                                                                              
      60 +   (`VClos`), constructor values (`VPack`), and primitive operators                                                                                                                                                  
      61 +   (`VPrim`) all share the same currying machinery in `apply`. `letrec` and                                                                                                                                          
      62 +   top-level mutual recursion both tie the knot using Haskell's laziness.                                                                                                                                            
      63 +3. **Print** (`app/Main.hs`) — evaluates `main`'s body in the global env and
